# COMP3011 Coursework 2 - Search Engine Tool

A Python search-engine tool that crawls
[quotes.toscrape.com](https://quotes.toscrape.com/), builds an inverted
index of every word on every page, and answers single- and multi-word
queries through an interactive command-line interface.

This project was developed for **University of Leeds COMP3011 - Web
Services and Web Data**, 2025-2026 academic year.

## Features

- **Polite breadth-first crawler** with a configurable politeness window
  (default: 6 seconds, as required by the coursework brief).
- **Inverted index** storing per-document term frequencies and token
  positions, persisted as a single JSON file.
- **Case-insensitive** search with conjunctive (AND) multi-word semantics.
- **Interactive CLI** providing the four assignment-mandated commands:
  `build`, `load`, `print`, `find`.
- **92 unit tests** with **100% line coverage** on every production
  module, using `requests-mock` for full network isolation.

## Architecture

```
   +-----------+       +------------+       +-----------+       +----------+
   |  Crawler  | --->  | Tokeniser  | --->  |  Indexer  | --->  |  Search  |
   |  (BFS,    |       | (HTML --> |       | (inverted |       | (print / |
   |  6 s gap) |       |  tokens)  |       |  index)   |       |  find)   |
   +-----------+       +------------+       +-----+-----+       +-----+----+
                                                   |                   |
                                                   v                   v
                                             data/index.json     CLI dispatch
                                              (JSON file)         (main.py)
```

Each module owns one responsibility and exposes a small public surface;
this keeps tests focused and design decisions easy to revisit.

## Usage

Start the interactive CLI from the **`app/`** directory:

```bash
python -m src.main
```

You should see:

```
Search Engine Tool - type 'help' for commands, 'exit' to quit.
> 
```

The four commands required by the brief are:

### `build`

Crawls [quotes.toscrape.com](https://quotes.toscrape.com/), builds the
inverted index, and saves it to `data/index.json`.

```
> build
Crawling https://quotes.toscrape.com/
(this may take several minutes due to the 6-second politeness window)
[1/200] fetching https://quotes.toscrape.com/
[2/200] fetching https://quotes.toscrape.com/login
...
Index built: 130 documents, 2480 unique terms.
Saved to /.../app/data/index.json
```

> Crawling the full site takes around **13 minutes** (130 pages times 6
> seconds). The politeness window is intentional and required by the
> coursework brief.

### `load`

Loads a previously built index from `data/index.json` into memory.
Useful for exploring the index without re-crawling.

```
> load
Loaded index: 130 documents, 2480 unique terms.
```

### `print <word>`

Prints the inverted-index entry for a single word - document frequency,
each posting's term frequency, and token positions. Position lists
longer than 10 are truncated for readability. `print` takes exactly one
word; extra arguments produce a warning and are ignored (use `find` for
multi-word queries).

```
> print love
Term: love
  Document frequency: 8
  Postings:
    doc 1 (https://quotes.toscrape.com/): tf=2, positions=[34, 102]
    doc 5 (https://quotes.toscrape.com/page/3/): tf=1, positions=[88]
    ...
```

A missing word prints a friendly message:

```
> print xyzabc
'xyzabc' is not in the index.
```

### `find <words...>`

Returns the URLs of pages containing **all** of the given words
(conjunctive / AND semantics). Each query word is run through the same
tokeniser used at build time, so capitalisation and punctuation are
handled consistently:

```
> find good friends
2 page(s) match:
  https://quotes.toscrape.com/page/2/
  https://quotes.toscrape.com/tag/friendship/

> find Good!
8 page(s) match:
  ...

> find indifference
1 page(s) match:
  https://quotes.toscrape.com/page/4/

> find unicorn
No pages match.
```

### `help` and `exit`

```
> help
Available commands:
  build              Crawl the website and build the inverted index.
  load               Load a previously built index from disk.
  print <word>       Print the inverted-index entry for <word>.
  find <words...>    Find pages containing all the given words.
  help               Show this message.
  exit               Quit the program.

> exit
```

You can also exit with **Ctrl+D** (EOF). **Ctrl+C** interrupts the
current input but keeps the REPL running, so an accidental
keypress doesn't lose your loaded index.

## Design decisions

The principal decisions and the lecture material that informed them:

### Inverted index with positions and term frequencies

Each posting stores both `tf` and the list of `positions`. Lecture 12
("Indexing") presents the trade-off: positions are not strictly required
to satisfy the coursework's `find` semantics, but they enable phrase
queries and proximity scoring later at no extra crawl cost. A 4-byte
integer `doc_id` (rather than embedding the full URL in every posting)
keeps inverted lists compact, with a single side table mapping IDs back
to URLs.

### Breadth-first crawl, single-domain restriction

The crawler implements the textbook BFS approach from Lecture 9
("Web Crawling"): a `frontier` queue (Python `deque`) and a `seen` set
for deduplication. The `seen` set is updated **at enqueue time** rather
than dequeue time so the queue cannot blow up on cyclic graphs (a
common quotes.toscrape.com pattern: every page links back to `/`).
Cross-domain links are filtered out by URL netloc comparison.

### 6-second politeness window via `time.monotonic`

`time.monotonic()` is used instead of `time.time()` so that the window
is immune to system-clock adjustments. The first request is **not**
delayed: `_last_request_at` initialises to `0.0`, so on the very first
fetch the elapsed time is large and the wait is skipped.

### JSON for index persistence

JSON is human-readable, easy for markers to inspect, and avoids the
forward-compatibility hazards of pickle. The minor pain point - JSON
disallows integer dict keys - is handled by stringifying `doc_id`s on
save and parsing them back on load.

### AND semantics for multi-word `find`

Lecture 13 ("Query Processing") describes conjunctive processing as the
default for major web engines because it matches user expectations for
queries like `find good friends` ("pages mentioning *both* good *and*
friends", not "either"). It is also computationally cheap with
`set.intersection`. Each query word is re-tokenised through the same
`tokenise()` used at build time, so `find Good!` works the same as
`find good`.

## Git workflow

This project uses a feature-branch workflow. Each piece of work happens
on its own branch and is merged back to `main` with `--no-ff` so the
branch topology is preserved in the history graph. Releases are marked
with annotated tags:

- **`v0.1-baseline`** - crawler, indexer, search, CLI integrated.
- **`v0.2-tested`** - 70 unit tests across the four core modules.
- **`v0.3-full-coverage`** - 92 tests, 100% line coverage.
- **`v0.4-docs`** - this comprehensive README.

```
git log --oneline --all --decorate --graph
```

shows the full history, including the merge commits.

## Generative AI declaration

**Tool used:** Claude Code, throughout the project, for refining function
implementations, debugging, and drafting documentation.

### Key value-adds

- **Architectural framing.** AI helped translate the lecture material on
  inverted indices into a concrete data-structure choice (per-posting
  `tf` + `positions`, integer doc-ids, JSON persistence) and articulate
  the trade-offs explicitly enough to defend in this README. This kept
  the design grounded in course content rather than improvised.
- **Test-isolation strategy.** The decision to use `requests-mock` and
  to monkeypatch `time.sleep` came out of a back-and-forth on how to
  test the politeness window without either taking 13 minutes or
  violating the brief's own 6-second rule. The resulting suite runs in
  under a second and exercises every error path.
- **Git-workflow discipline.** The feature-branch + `--no-ff` merge +
  annotated-tag pattern was suggested as a way to make the development
  history legible to a marker. I followed and adapted it; the resulting
  graph (four feature branches, multiple tagged releases) is more
  professional than what I would otherwise have produced under time
  pressure.
- **Documentation density.** Sections like "Design decisions" were
  iterated with AI to keep them tight (one paragraph per choice, with a
  lecture reference) rather than turning into a wall of text.

### Where I pushed back

- An early suggestion was to use `\w+` for tokenisation. I pointed out
  this matches non-ASCII (CJK) characters and would inflate the index
  on stray content, so we switched to `[a-z0-9]+` and documented the
  apostrophe trade-off explicitly.
- AI initially proposed a strict `Content-Type` check that rejected
  responses lacking the header. The first round of crawler tests
  uncovered the issue (every `requests-mock` response failed), which
  drove me to surface the assumption in tests rather than weaken the
  check in production code.
- For the GenAI declaration itself, the first draft was an exhaustive
  blow-by-blow log. I rewrote it to focus on outcomes and critical
  thinking, which is what the brief actually rewards.

### Reflection

Using Claude Code accelerated the boilerplate (tests, docs, refactors)
without obscuring the algorithmic core, which I still had to read,
explain, and verify line-by-line. The largest learning gain came from
watching AI propose a solution, then having to identify *why* it was
sometimes wrong - that gap between "plausible" and "correct" is where
the actual understanding lives. The biggest risk I tried to manage is
**outsourcing comprehension**: if I cannot explain a function in my own
words, I shouldn't ship it, regardless of whether the tests pass.
